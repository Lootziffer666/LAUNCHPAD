# 3D Habitat World with model-viewer boxes

The LAUNCHPAD desktop Electron app gains a new full-screen "Habitat World" mode: a 3x2 grid of 3D boxes, each with a door animation and a `<model-viewer>` GLB scene behind it. Six tiles (play, apps, library, create, watch, explore) map to six GLB models (kitchen, office, cabin, stanley_office, night_sky, forest). Entry is via Ctrl+H or the new "3D Welt" tile on the launchpad home screen. The prototype's single-model-viewer-for-play-only restriction is lifted to cover all 6 boxes.

Watch for: ~23 MB of GLB assets committed to the repo with no lazy loading or compression strategy (confirmed); timer-based state transitions create a race condition when Escape is pressed mid-animation (confirmed); the "3D Welt" tile button passes an origin argument to a handler that ignores it (confirmed).

## High-level view

HabitatWorld.jsx is a self-contained functional component with all six box definitions inlined as static constants. It uses CSS 3D transforms for the camera fly-in and `@google/model-viewer` (web component, side-effect imported) for GLB rendering inside the active box. The four-value `view` enum (`home | opening | in | closing`) drives both the camera position and the door animation, but the `setTimeout`-based transitions between states are uncancellable, creating a race when a user presses Escape during the opening phase.

All eight GLB files (23 MB total) and diorama/tile images (3.3 MB) are committed directly to git and bundled into the Electron app's resources. There is no code-splitting or lazy loading — model-viewer loads eagerly when the overlay appears. For an Electron desktop app this is acceptable at runtime, but the git history bloat is permanent.

Two unrelated fixes (duplicate `recoveryCode` state in `Parental.jsx`, `submitRecovery` rename in `WindowsDesktop.jsx`) shipped in the same set of commits.

## Timer race in open/close state machine

`openBox` fires `setTimeout(() => setView('in'), 980)` and `closeBox` fires `setTimeout(() => { setView('home'); setActive(null); }, 1050)`. Neither stores the timer ID. If the user presses Escape during the opening animation (which calls `closeBox` immediately), the first timeout still fires and sets `view` to `'in'` after 980 ms — potentially after the closing timeout has already returned the component to `'home'`. The resulting state sequence is `opening → closing → home → in`, which renders the 3D overlay (`show3d = active != null && view === 'opening' || view === 'in'`) with `active` already nulled by the closing callback.

A single ref (e.g. `timerRef.current = setTimeout(...)`) cleared at the top of both `openBox` and `closeBox`, plus a `useEffect` cleanup, eliminates the race.

## Asset weight and repository impact

The eight GLB files in `desktop/public/models/` total 23 MB:

```
kitchen.glb      7.3 MB
cat_a.glb        4.8 MB
forest.glb       3.3 MB
office.glb       3.6 MB
cat_b.glb        2.5 MB
night_sky.glb    980 KB
stanley_office   560 KB
cabin.glb        332 KB
```

For a desktop Electron app bundled with the installer this is acceptable at runtime — assets load from local disk. The concern is git history: these binaries are not diffable, and every revision doubles the storage. Git LFS would decouple asset weight from clone time. The cat models (`cat_a.glb`, `cat_b.glb`) are used by the prototype's companion cats layer (updated in the same diff), not by HabitatWorld directly.

## Keyboard shortcut collision surface

Ctrl+H is registered via a global `keydown` listener in `App.jsx`. The handler fires regardless of whether an overlay (PIN gate, app shell) is currently active. Opening habitat while the PIN gate is displayed layers habitat underneath (z-index is lower), so there's no visual glitch, but the mode state silently changes. If the user then dismisses the gate, they land in habitat mode rather than the expected launchpad. Gating the shortcut on `!app && !gate` would prevent silent mode switches.
